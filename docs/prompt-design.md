# 剧本杀 AI 选角助手 — Prompt 设计文档

> 设计原则: 不展示完整 Prompt，阐述每段的设计意图与取舍逻辑

---

## 1. System Prompt 总体结构

```
┌─────────────────────────────────┐
│ 1. 角色定义 (你是谁)               │  建立 Agent 身份，设定行为边界
├─────────────────────────────────┤
│ 2. 能力声明 (你能做什么)            │  告知可用工具，防止 LLM 自行编造能力
├─────────────────────────────────┤
│ 3. 工作原则 (你怎么做)              │  5 条硬约束，覆盖证据/追问/输出
├─────────────────────────────────┤
│ 4. 典型流程 (你该怎么做)            │  3 条示例路径，few-shot 行为引导
├─────────────────────────────────┤
│ 5. 输出格式 (你说成什么样)          │  规定回复必须包含的结构
└─────────────────────────────────┘
```

---

## 2. 各段设计意图

### 2.1 角色定义

**设计意图**: 让 LLM 进入"专业选角助手"角色而非通用 chatbot。关键措辞：

- "专为剧本杀玩家推荐角色的智能 Agent" — 限定领域
- 列出可用工具清单 — 划定能力边界，防止 LLM 声称能做搜索之外的事

**设计取舍**: 没有用"你是 AI 助手"这类泛化表述，而是用领域限定词 `剧本杀AI选角助手` 让 LLM 在回答时更精准地使用行业术语。

### 2.2 工作原则（5 条硬约束）

| # | 原则 | 设计意图 |
|---|------|---------|
| 1 | **证据优先** | 对抗 LLM 最核心的风险——幻觉。明确要求"绝不胡编角色" |
| 2 | **多维度驱动** | 防止 LLM 只看 MBTI 忽略历史记录和追问答案 |
| 3 | **主动追问** | 信息不足时不强行推荐，比"差不多就行"更符合产品预期 |
| 4 | **循序渐进** | 给定标准 SOP 路径，防止 LLM 跳步 |
| 5 | **完整输出** | 强制结构化的推荐结果，防止遗漏风险提示 |

### 2.3 典型对话流程（隐式 few-shot）

**设计意图**: 不在 System Prompt 里塞完整对话示例（浪费 token），而是用流程描述做行为引导：

```
用户说"帮我选告别诗的角色"
→ 你先 retrieve_knowledge("告别诗") 了解角色
→ 再 analyze_user_persona 分析用户画像
→ 用 recommend_roles 给出推荐
→ 如果用户犹豫，用 compare_roles 做对比
→ 如果用户只说了类型没说具体剧本，先用历史记录推断偏好，再追问
```

**设计取舍**: 这比写 3 组完整对话示例节省约 60% token，但同样能达到行为约束效果。

### 2.4 输出格式约束

```markdown
🎭 角色推荐 + 匹配分数
✅ 推荐原因（特征匹配、情感线匹配等）
⚠️ 注意事项（雷点、反串等）
📋 备选方案
```

**设计意图**: 用 emoji 作为视觉锚点，让 LLM 的输出结构稳定可解析，同时读起来自然。

## 3. 工具描述 (Function Definitions) 设计

### 3.1 设计原则

1. **描述即约束**: 每个 tool 的 description 里写明能力边界
   ```
   "检索结构化信息。包含角色特征、情感线、适合玩家类型等。"
   ```
   这样 LLM 才不会用 search_script 去做角色推荐。

2. **参数限定枚举**: 能枚举的不用自由文本
   ```
   gender: ["男", "女", "不限"]
   crossGender: ["no", "maybe", "yes"]
   ```
   减少 LLM 输出非预期值的概率，也降低下游代码的容错成本。

3. **required 参数精确**: 只标记真正必须的参数
   - search_script 只需要 scriptName（其他信息可以从知识库补）
   - analyze_user_persona 所有参数都 optional（用户可能只填 MBTI 不填星座）

## 4. 搜索 Prompt 设计

**场景**: `searchAgent.js` 中当使用 LLM 替代 Bing 时的 Prompt

**核心思路**: 不让 LLM 做"搜索"，而是让它**回忆训练数据中见过的公开测评**。

```
"请回忆你在训练数据中见过的、来自小红书、迷圈、千岛、抖音等社交平台上关于剧本《X》的公开帖子和测评内容。"

严格要求:
1. 只基于记忆中真实的社交平台公开内容回答，绝对不要编造角色
2. 如果你确实没见过或不确定，直接返回 {"sources":[]}
3. 排除 NPC、DM、主持人、店家、作者等非玩家角色
4. 每个角色说明性格特征、情感线类型、适合的玩家、可能的雷点
```

**设计取舍**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| 让 LLM 编造搜索结果 | 永远有结果 | 不可信 |
| 只用 Bing API | 可信 | 需要付费、可能搜不到 |
| LLM 回忆 + Bing 双通道 | 互补 | 需要处理 LLM 的"不确定" |

当前用双通道：先试 LLM 回忆（免费），失败再 Bing。LLM 的 critical constraint 是"不确定就直接返回空"，这是对抗幻觉的关键。

## 5. 角色提取 Prompt 设计

**场景**: `roleProfiler.js` 中从搜索结果提取角色的 LLM Prompt

**核心约束**:
```
"只从来源文本中提取角色，不要编造"
"排除明确标注为 NPC、DM、主持人、店家、作者的角色"
"输出严格 JSON 格式"
```

**设计意图**:
- Temperature 降到 0.1（极致确定性）
- 回退机制: LLM 提取失败 → 自动回退正则模式
- JSON 输出用正则 `/\{[\s\S]*\}/` 做容错提取，防止 LLM 在 JSON 前后加废话

---

## 6. 追问生成策略

**设计思路**: 追问不依赖 LLM，而是**基于已识别的角色差异自动生成**。

| 触发条件 | 生成的追问 |
|----------|-----------|
| 角色间情感线差异大 | "你这次最想体验哪类情感线？（爱情/亲情/友情/...)" |
| 角色间性格差异大 | "你更容易代入哪种表达方式？（隐忍克制/主动外放/...)" |
| 角色有雷区标签 | "你希望避开哪些内容？（卑微感/原生家庭痛点/...)" |
| 性别冲突 | "当前资料中存在与你性别偏好不一致的角色，是否接受反串？" |

**为什么不用 LLM 生成追问**: 追问需要**100% 基于已识别的真实角色信息**，用代码逻辑比 LLM 更可靠，且零延迟。

---

## 7. 关键设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| 追问用代码 vs LLM | 代码 | 可靠性，零 token 成本 |
| 角色提取用 LLM vs 正则 | 双模式 | LLM 优先（准），正则兜底（稳） |
| Temperature | 0.1-0.3 | 推荐场景不需要创造性，需要稳定性 |
| 搜索用 LLM vs API | 双通道 | LLM 免费但可能不确定，Bing 付费但可靠 |
| 工具参数枚举 vs 自由文本 | 枚举 | 减少下游容错成本 |
